## Priorisierte Findings

### P0 — CSRF auf Admin-POST-Routen erlaubt erzwungene State-Changes trotz Confirm-Gate

**Risiko:** Hochkritisch. Ein eingeloggter Admin kann über eine fremde Webseite unbemerkt zu POST-Requests auf `/admin/...` verleitet werden (klassische Cross-Site Form Submission). Die betroffenen Endpunkte ändern Systemzustand (User löschen, Backup restore, License austauschen, Mesh-Key rotieren etc.), aber haben **kein** serverseitiges CSRF-Decorator/Gate. Die zusätzliche Confirm-Gate-Abfrage schützt hier nicht zuverlässig, weil das erwartete Token statisch ist (`CONFIRM`) und im Frontend/offenem Quelltext sichtbar ist.  

**Evidence:**
- Statisches Confirm-Gate-Token im Backend: `RISK_CONFIRM_TOKEN = "CONFIRM"`.【F:app/routes/admin_tenants.py†L48】
- Mehrere sensitive POST-Endpunkte ohne `@csrf_protected`, z. B.:
  - `/settings/users/delete`.【F:app/routes/admin_tenants.py†L337-L346】
  - `/settings/license/upload`.【F:app/routes/admin_tenants.py†L422-L430】
  - `/settings/backup/restore`.【F:app/routes/admin_tenants.py†L513-L523】
  - `/settings/mesh/rotate-key`.【F:app/routes/admin_tenants.py†L553-L559】
- Frontend verrät/erzwingt denselben statischen Wert `CONFIRM` (leicht in CSRF-Form nachbaubar).【F:app/templates/settings.html†L101-L103】【F:app/templates/settings.html†L145-L147】【F:app/templates/settings.html†L250-L253】

**Repro (minimal):**
1. Als Admin in KUKANILEA eingeloggt bleiben.
2. Externe HTML-Seite öffnen, die automatisch ein Formular an `https://<ziel>/admin/settings/mesh/rotate-key` postet, inkl. Feld `confirm=CONFIRM`.
3. Request wird serverseitig akzeptiert (nur Login + Rolle + statischer Confirm-String).
4. Kritische Aktion wird ohne legitimen Origin-CSRF-Nachweis ausgeführt.

**Empfohlene Behebung:**
- Auf **allen** mutierenden `/admin`-POST-Routen serverseitig `@csrf_protected` aktivieren.
- Confirm-Gate nicht als statischen Shared-String nutzen (mind. per-request nonce/challenge binden, besser: serverseitiges re-auth step-up bei hochriskanten Aktionen).
- Optional zusätzlich `Origin/Referer` Validierung und `SameSite=Strict` für Session-Cookies härten.

### P1 — Confirm-Gate-Matrix ist nicht zentral/erzwingbar im Agent-Executor (Policy Drift)

**Risiko:** Hoch. Laut Tool-Matrix dürfen Confirm-Gate-Tools niemals autonom laufen. In der Runtime wird das jedoch nicht zentral im Executor/Registry erzwungen; der Executor führt das Tool direkt aus. Damit hängt Security-Verhalten an UI-Konventionen/Agent-Disziplin statt an einer harten Laufzeitkontrolle.

**Evidence:**
- Spezifikation fordert harte Confirm-Gate-Enforcement für markierte Tools.【F:app/agents/config/TOOLS.md†L5-L16】
- Executor ruft Tool direkt auf, ohne Confirm-Flag/Challenge-Check.【F:app/agents/executor.py†L13-L24】
- Registry hält nur Tool-Objekte, keine Sicherheitsmetadaten/Policy-Hooks für Confirm-Gates.【F:app/tools/registry.py†L9-L23】

**Repro (konzeptionell, aus Code ableitbar):**
1. Ein als riskant klassifiziertes Tool wird in die Registry aufgenommen.
2. Planner/Agent liefert Tool-Action.
3. `AgentExecutor.execute()` ruft `tool.run(**params)` ohne serverseitige Confirm-Verifikation auf.
4. Aktion kann autonom durchlaufen, obwohl Matrix „Confirm-Gate=True“ fordert.

**Empfohlene Behebung:**
- Confirm-Gate als **verpflichtendes Runtime-Policy-Attribut** pro Tool in Registry-Modell aufnehmen.
- Executor muss bei `requires_confirm=True` serverseitig blocken, solange kein gültiges, kurzlebiges Approval-Token im Request-Kontext vorliegt.
- Unit-Tests: „risky tool without confirm -> denied“.

### P2 — CSP ist vorhanden, aber durch `unsafe-inline` in Script/Style deutlich aufgeweicht

**Risiko:** Mittel. CSP existiert, verhindert aber durch `script-src 'unsafe-inline'` viele XSS-Schutzwirkungen. Bei künftigem Template/DOM-Injection-Bug wäre Exploit-Barriere deutlich geringer.

**Evidence:**
- Aktiver CSP-Header enthält `script-src 'self' 'unsafe-inline'` und `style-src 'self' 'unsafe-inline'`.【F:app/__init__.py†L153-L160】
- Parallele Security-Header-Funktion mit gleicher Aufweichung bestätigt das Muster.【F:app/security.py†L19-L25】

**Repro (policy-level):**
1. Erzeuge (oder finde) eine HTML/JS-Injection in einem Template/DOM-Sink.
2. Inline-Payload wird durch aktuelle CSP nicht grundsätzlich blockiert.
3. Angriff hat höhere Erfolgswahrscheinlichkeit als bei nonce/hash-basierter CSP.

**Empfohlene Behebung:**
- Schrittweise Migration auf Nonce/Hash-basierte CSP ohne `unsafe-inline` für Scripts.
- Inline-Skripte in dedizierte statische JS-Dateien auslagern.

## Ergebnis je Prüfauftrag

1. **Zero-CDN Scan (produktive Dateien):** Keine produktiven externen CDN-Includes gefunden (Scripts/CSS in Layout lokal eingebunden).【F:app/templates/layout.html†L7-L20】
2. **CSP/Guardrails:** CSP vorhanden, aber Script/Style-Policy zu permissiv (siehe P2).【F:app/__init__.py†L153-L160】
3. **Secrets/Token-Leaks:** Keine klaren Live-Secrets/Hardcoded API-Keys im produktiven Code identifiziert; Hauptthema ist statischer Confirm-String statt Secret-Leak (siehe P0).【F:app/routes/admin_tenants.py†L48】
4. **Confirm-Gate-Matrix:** Dokumentierte Anforderungen existieren, aber zentrale technische Enforcement-Lücke (siehe P1).【F:app/agents/config/TOOLS.md†L16】【F:app/agents/executor.py†L24】

## Offene Annahmen
- Bewertung basiert auf statischer Codeanalyse im Repo-Stand (ohne laufende Browser-Session mit Admin-Cookie).
- Falls vorgelagerte Infrastruktur zusätzliche CSRF-Defense erzwingt (z. B. WAF mit Origin-Policy), reduziert das Ausnutzbarkeit, ersetzt aber keine serverseitige Applikationskontrolle.
